# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

Personal dotfiles managed with GNU Stow. Top-level package directories (`nvim/`, `zsh/`, `git/`, `tmux/`, `bin/`) mirror `$HOME` layout and are symlinked into place by `stow`. `install/` holds per-tool bootstrap scripts. `windows/` and `windows_setup.ps1` are for the Windows laptop (symlinks the Windows Terminal `settings.json` and installs the `lemonade server` half of the `$BROWSER` setup â€” see below).

## Common commands

```sh
./install.sh              # stow only (default)
./install.sh list         # list available installers
./install.sh <name>       # run one installer, then stow
./install.sh all          # run every installer, then stow
./install.sh all-confirm  # prompt Y/n per installer, then stow
./install.sh missing      # run every installer non-interactively (REINSTALL=0), installing only what's absent
REINSTALL=1 ./install.sh <name>  # force reinstall a tool that's already present
```

The stow step always runs at the end and links `nvim zsh git tmux bin zk starship` into `$HOME` (plus `claude` on non-`home` envs). The `bin/` package holds standalone helper scripts that land on `$PATH` via `bin/.local/bin/` (e.g. `browser-open`, the `$BROWSER` wrapper, and `open-file`, which opens an NFS path in the laptop's Explorer â€” both route by env and share the lemonade tunnel; see below). The `zk/` package holds the global zk config (`zk/.config/zk/` â†’ `~/.config/zk/`); see the **work-vault** section. The `starship/` package holds the prompt config (`starship/.config/starship.toml` â†’ `~/.config/starship.toml`); it's intentionally minimal â€” relies on starship's built-in default prompt, just setting `command_timeout` to keep `git status` from hanging on large repos. starship itself is installed by `install/zsh/install.sh`.

## DOTFILES_ENV

First run prompts for an environment and writes the choice to `~/.dotfiles_env`. Valid values: `work-argos`, `work-devcompute`, `home`. The variable is exported before installers run, sourced by `.zshrc`, and read by `nvim/.config/nvim/init.lua`. Behavior gated on it:

- `install/lib.sh:ensure_brew` â€” Linuxbrew is bootstrapped/used only when `use_brew` returns true (currently `home`). `try_brew` returns 1 elsewhere so installers fall back to `eget`.
- `.utils.sh` (repo root, non-stowed) defines `use_brew()` â€” the canonical brew-vs-fallback gate. Sourced by both `install/lib.sh` and `zsh/.zshrc` so installers and the live shell agree. Use it instead of inlining the env check.
- `.utils.sh` also defines `workspace_dir()` â€” the canonical big-disk root for the env (home is small on work hosts): `/local/mnt/workspace/juliray` (devcompute), `/prj/qct/mlsys/markham/scratch/juliray` (argos), `$HOME` elsewhere. Park large/regenerable data here and derive specific paths from it rather than re-checking the env. Built on it: `nvm_dir()` â†’ `$(workspace_dir)/.nvm` (used by `install/nvm/install.sh` and `zsh/conf.d/nvm.zsh`); `zsh/conf.d/python.zsh` exports `PIP_CACHE_DIR`, `UV_CACHE_DIR`, `UV_PYTHON_INSTALL_DIR` under `$(workspace_dir)` to keep pip/uv off home. Its sibling `scratch_dir()` is the shared **NFS** scratch root (network-visible, vs `workspace_dir` which may be host-local): same path as `workspace_dir` on argos, but the lasvegas NFS mount (`/prj/qct/mlsys/lasvegas/scratch/juliray`) on devcompute. It backs the `scratch` alias in `zsh/env/<env>.zsh` and `ptrace`'s HTTP-mode file-server root â€” use it instead of re-hardcoding the scratch path per env.
- `.utils.sh` also defines `work_vault_dir()` (â†’ `$(workspace_dir)/work-vault`, override `$WORK_VAULT_DIR`) and `work_vault_remote()` (â†’ a private personal repo on GHE, `github.qualcomm.com/juliray/work-vault`, override `$WORK_VAULT_REMOTE`) â€” the seams for the **work-vault** (see its own section below). Lives under `workspace_dir` so the vault stays off the small home partition on work hosts; on the laptop / any `home` env `workspace_dir` is already `$HOME`, so it lands at `$HOME/work-vault` there. The local path may thus differ per host â€” harmless, since the git remote (not the path) is what ties every clone together. `zsh/conf.d/vault.zsh` exports `WORK_VAULT_DIR` from the seam so the `vault` helper, the slash commands, and the `vault-explorer` agent all resolve the same path.
- `zsh/env/<env>.zsh` is sourced after `conf.d/*.zsh` for env-specific paths/aliases.
- `git/.gitconfig.<env>` is symlinked to `~/.gitconfig.local` (included by `git/.gitconfig`).
- The `ssh/` package (`ssh/.ssh/config`) is stowed on the work envs only (alongside `claude`), so the repo owns the work `~/.ssh/config`. Because that path is normally a real hand-edited file, `install.sh` moves any pre-existing non-symlink to `~/.ssh/config.pre-stow.bak` before stowing. It carries the `lemonade` `RemoteForward 2489` block (the `$BROWSER` tunnel â€” see below); the laptop's equivalent is written by `windows_setup.ps1`, not stowed.
- `init.lua` configures `clangd` to run via `./run-in-docker` on the work envs, plain `clangd` on home.
- `install/claude-code/install.sh` runs only on the work envs (it pulls Claude Code from Qualcomm's internal qgenie installer and registers the Tavily search MCP, both corp-network-only). It gates with an explicit `case "${DOTFILES_ENV:-}"` rather than a `.utils.sh` helper, since the condition (work-only) doesn't match `use_brew`'s home-only sense â€” follow this `case` pattern for other work-only installers.
- `install/work-vault/install.sh` also runs only on the work envs (same `case` pattern â€” the vault remote is a private repo on GHE, `github.qualcomm.com`, corp-network-only). Clones/creates the work-vault repo to `$(workspace_dir)/work-vault` â€” see the **work-vault** section below.

When adding env-conditional logic, follow these existing seams rather than introducing new env checks.

## Installer conventions (`install/<name>/install.sh`)

Every installer sources `install/lib.sh` and follows the same shape. Use the helpers â€” don't reinvent them:

- `should_install <cmd> [version-cmd...]` / `should_install_path <name> <path>` â€” gates the install. Honors `REINSTALL=1`; silently skips when stdin isn't a TTY.
- `try_brew <formula>` â€” installs via brew on `home`, returns 1 elsewhere so callers fall back. brew failures are fatal (treated as a config bug, not transient).
- `ensure_eget` + `eget_install <gh-repo> --to "$BIN_DIR"` â€” the standard non-brew fallback. The wrapper anti-matches `.deb`/`.rpm`/`.apk` so eget doesn't prompt; pass extra `--asset` flags to disambiguate further (e.g. gnu vs musl, arm64 vs x86_64). Binaries land in `$HOME/.local/bin`.
- `info` / `error` for logging; `error` exits non-zero.

All installer scripts run with `set -euo pipefail`.

New `install/<name>/install.sh` files must be `chmod +x` â€” the dispatcher in `install.sh` only picks up executable installers (silently skips otherwise).

`install/claude-code/install.sh` shows two non-binary-install patterns worth reusing: (1) it registers a remote MCP server (`claude mcp add --scope user`) whose config lands in `~/.claude.json` â€” not a stowable file, so the installer *is* the version-controlled recipe; (2) the Tavily key is stored as the literal `${TAVILY_API_KEY}` (single-quoted so the shell doesn't expand it) for Claude to expand at runtime, keeping the secret out of `~/.claude.json` â€” it lives only in `~/.zsh_secrets`. MCP idempotency uses `claude mcp get <name>` (the binary may need `hash -r` first on a fresh install, since bash caches command lookups).

`install/perfetto/install.sh` is another work-only installer (same `case "${DOTFILES_ENV:-}"` gate â€” trace viewing is a work-host need). It shows the "download is not a GitHub release, so not eget" pattern: `get.perfetto.dev/trace_processor` is a self-caching Python wrapper (the ~14MB real binary lands under `~/.local/share/perfetto/prebuilts/` on first run), so the installer `curl`s the wrapper into `$BIN_DIR` and does one warm-up run (`hash -r` first) to cache the binary at install time. See the `ptrace` helper (Zsh layout) for the end-to-end trace-viewing flow.

## Zsh layout

`zsh/.zshrc` is intentionally tiny. It loads, in order: secrets (`zsh/.zsh_secrets`, gitignored) â†’ antidote plugins (`.zsh_plugins.txt`) â†’ every `conf.d/*.zsh` fragment â†’ the matching `env/<DOTFILES_ENV>.zsh` â†’ `starship init`. Add new shell behavior as a fragment in `conf.d/` (general) or `env/` (env-specific), not inline in `.zshrc`.

`zsh/conf.d/notify.zsh` defines `notify [msg]` â€” POSTs a message to an [ntfy](https://ntfy.sh) topic so a long command can ping your phone/desktop when it finishes (`some-long-build; notify "build done"`). It captures `$?` first, so the title/priority reflect the preceding command's exit status. Config via `~/.zsh_secrets` (gitignored): `NTFY_TOPIC` (required) and `NTFY_URL` (defaults to the public `https://ntfy.sh`). The cross-environment design rationale â€” why ntfy over OSC escapes / Gotify / chat webhooks, and the eventual click-to-navigate-back plan â€” lives in `notes/notifications.md`.

`zsh/conf.d/perfetto.zsh` defines `ptrace [-w|-n] <trace-file> [port]` â€” views a trace in the Perfetto UI, with **two modes** picked automatically by file size (cutoff `$PTRACE_HTTP_MAX`, default 1800 MB; `-w` forces HTTP/web, `-n` forces native, `$PTRACE_MODE` sets the default; `ptrace -k` kills every ptrace-managed server on the host). Both self-reap on `$PTRACE_IDLE_TIMEOUT` inactivity (default `15m` for HTTP, `8h` for native, `never` to disable) and best-effort open the URL via `$BROWSER`.
- **HTTP mode** (small traces, the common case): starts an **ephemeral, single-file** `python3` server (bound `0.0.0.0`) that serves **only this trace's basename** plus a tiny launcher page at `/__ptrace_open__` (every other path â†’ 404, so no sibling files under the dir are exposed). The launcher drives Perfetto's **postMessage** protocol â€” `window.open`s the UI, fetches the trace from this same-origin server as an ArrayBuffer, and posts it into the UI tab (PING/PONG handshake first) â€” so the browser parses it client-side (WASM). postMessage needs **neither a CSP flag nor CORS** (and sidesteps the mixed-content block an HTTPS UI hits fetching an `http://` trace) â€” unlike the old `?url=` deep link it replaced. The file crosses the wire and the browser's ~2 GB ceiling applies (hence the size cutoff). The URL points at **this host by name** (`$(hostname)`, override `$PTRACE_HTTP_HOST`), so the laptop reaches it directly over the corp network â€” **no ssh tunnel**. The server **reaps itself** ~`$PTRACE_HTTP_GRACE` (default 30s) after the launcher's fetch delivers the trace (the delivery signal; a short idle fallback reaps an orphan if the trace is never fetched), so a trace is on the wire only briefly for one recipient's click â€” this **single-file + reap-on-delivery** model replaced an earlier one-server-per-root design that exposed the whole `scratch_dir`/`workspace_dir` tree for 8h (the security fix for the **ptrace-sharing** goal â€” see the memory note). The port is **per-user** (base `$PTRACE_HTTP_PORT`, default 9201, `+ UID%50`, then probed upward for a free one) so users on a shared host don't collide; viewing N traces = N short-lived servers/tabs.
- **Native mode** (large traces, or `-n`): runs `trace_processor`'s HTTP RPC "native accelerator" (installed by `install/perfetto/install.sh`) with the trace preloaded, so parsing happens on the work host and only rendered results cross the wire â€” no size limit. But it's one server = one trace = one port: re-running on the same port **swaps** the trace (reaps the prior server), a different port gives a concurrent server/tab, and viewing N traces needs N ports/tunnels. It prints a **version-pinned** UI URL (`ui.perfetto.dev/vXX.Y-hash/#!?rpc_port=<port>`, the `vXX.Y-hash` read live from `trace_processor --version` so the UI's wire protocol matches the server). Two non-obvious gotchas: (1) the wrapper execs the real binary as `trace_processor_shell-<hash>`, so the reap `pkill` pattern must span that suffix (a bare `trace_processor server` misses it); (2) **the CSP constraint** (learned the hard way) â€” unlike HTTP mode, native mode's RPC websocket only ever connects to `127.0.0.1:<port>` *on the laptop*; binding to `0.0.0.0` + browsing `hostname:<port>` shows the status page but will *not* load a trace, so an `ssh -L <port>:localhost:<port> <workhost>` tunnel is **mandatory** here (the helper prints it). Native mode additionally needs the one-time YES on the "Trace Processor native acceleration" dialog.

## work-vault (Claude working-context + notes)

A git-synced zk notebook that doubles as Claude's structured working-context store and a home for regular notes. It exists to solve two coupled problems: scattered, undiscoverable handoff markdown, and main-context bloat when several explorations finish. **Three stores, clean boundary:** the vault (live, connected working-context) vs. `notes/` (decisions about *this* repo, committed with it) vs. Claude `memory/` (durable atomic facts).

- **Repo + location.** Dedicated private repo `github.qualcomm.com/juliray/work-vault` on GHE â€” the corp-hosted platform, which is the CCI-correct home for the vault's Yellow/Red content (external `github.com` cloud is not). A *personal* repo is fine while the vault is single-user; CCI requires a GHE org + N2K team only if it must be shared with multiple users (a personal repo can't attach an N2K team). Cloned to `$(workspace_dir)/work-vault` on each work host (off the small home partition) and to `$HOME/work-vault` on the laptop / `home` env (where `workspace_dir` is `$HOME`). Created/cloned by `install/work-vault/install.sh` (work-gated). Seams: `work_vault_dir()` / `work_vault_remote()` in `.utils.sh`, `WORK_VAULT_DIR` and `ZK_NOTEBOOK_DIR` exported by `zsh/conf.d/vault.zsh`.
- **Layout** = flat project directories, indexed by zk: `index.md` (top MOC) â†’ `projects/<slug>/index.md` (per-project MOC) â†’ linked `projects/<slug>/<topic>.md` notes. A note's **kind** (exploration / plan / task) is a frontmatter **tag** (`tags: [plan]`), not a subdirectory â€” so a note's kind is fluid (retag, don't move â†’ wikilinks never break), tags compose, and `zk list --tag plan` / the `all-{explorations,plans,tasks}` named filters give cross-project views. Freeform personal notes live at the vault root (untagged) and wikilink into the same graph. The only required frontmatter is the kind tag (template-generated, never hand-typed); everything else stays plain â€” the first paragraph is the "lead" (gist), the `# heading` is the title, the directory determines the project.
- **zk (`.zk/`)** â€” Go CLI + built-in LSP server (`zk lsp`), installed at the end of `install/work-vault/install.sh` (there is no `install/zk/`). The zk config (`config.toml` + `templates/`) lives in the dotfiles **`zk` stow package** â†’ `~/.config/zk/`, which zk reads as its *global* config and every notebook inherits â€” so the vault itself keeps only an **empty `.zk/` marker dir** (zk recognizes a notebook only by the presence of `.zk/`; config/templates resolve from the global dir). Edit `zk/.config/zk/config.toml` in dotfiles, not the vault. `config.toml` defines groups (exploration, plan, task, project â€” the three kind groups all share `paths = ["projects"]` and one `kinded.md` template, differing only in the name the note-creation alias selects via `-g`), wiki link format, the `all-*` named filters, and **aliases**. Note creation, search, relations, and MOC-ensure are zk aliases (not `vault` subcommands): `zk exploration|plan|task "<title>"` (create a flat kinded note in the current project â€” slug from `$ZK_CWD`, MOC ensured, `--extra` feeds the template's `tags:`; a **body piped on stdin** is captured into the note via the template's `{{content}}`, so `printf 'body' | zk plan "Title"` is a one-shot create-and-write â€” the `-i` flag is added automatically when stdin isn't a tty, leaving the interactive nvim flow untouched), `zk find <query>` (full-text â†’ `path<TAB>title<TAB>lead` lines; for inline reading of a single match prefer `vault find`), `zk related <path>` (`--related` âˆª unlinked mentions, MOC rows dropped), `zk moc <slug>` (idempotent project-MOC ensure). The composed logic lives in zk, not bash. `~/.config/zk/templates/` holds the Handlebars templates (`default.md`, `kinded.md`, `note.md`, `task.md`, `project-moc.md`); the kind templates render `{{content}}` (the stdin body, empty in the interactive flow). The SQLite index (`notebook.db`, gitignored, lives in the vault's `.zk/`) is rebuilt per-host via `zk index`. zk-nvim provides Neovim integration (completions on `[[`, go-to-definition, dead-link diagnostics, backlinks).
- **`bin/.local/bin/vault`** â€” thin CLI holding only what zk has no native equivalent for: `dir` (print vault path, error if absent), `slug` (derive project key from cwd's git remote, fallback dir name â€” the seam the `zk slug`/note-creation aliases call back into), `find [-p] <query>` (locate a note by **title or filename** substring â€” *not* a full-text body search, which would match far too much; query and haystack are both normalized so `zk mcp`/`zk-mcp`/`ZK MCP` all match, and MOC `index.md` rows are dropped. A **single** match prints the note **body** to stdout for inline reading â€” the `@`-mention resolver that saves a Read round-trip; `>1` prints `path<TAB>title` lines; `0` exits 1; `-p` forces the path list even on a single match), `reindex` (regenerate every MOC's link lists + the top index via `zk list` queries; cd's to the vault so its cwd-relative root-notes query is stable), `sync "<msg>"` (reindex, then add+commit+push, push failure non-fatal). Note creation / search / relations / MOC-ensure moved to zk aliases (see the zk bullet above).
- **Script-managed navigation.** Each MOC and `index.md` has a single `<!-- auto:content -->` â€¦ `<!-- /auto:content -->` region that `reindex` rewrites in full (section headings + bullet lists). `reindex` runs one `zk list --tag <kind>` query per bucket (`## Explorations`/`## Plans`/`## Tasks`, plus a `## Notes` bucket for notes carrying none of those tags), formatting each bullet straight from zk: `{{link}}` yields the wikilink (honoring the notebook's link-format, so no path/stem reconstruction in bash) and `{{lead}}` (first paragraph, newline-collapsed + first-sentence-trimmed) is the bullet text â€” a heading appears only when that bucket is non-empty. Prose outside the markers is preserved. The `## Tasks` bucket is special: it's rendered by `render_tasks` in `vault` â€” a bash loop over `zk list`, not a `bullets-*` format alias â€” because each task expands to a parent bullet plus its **open sub-tasks** (body checkboxes, re-indented) and a `(done/total)` tracker. At the **top `index.md`** that bucket also groups tasks under a `### <project>` sub-heading and shows **open tasks only** (`status: done` hidden â€” the top index is an actionable dashboard); per-project MOCs list every task flat. `sync` reindexes first, so every save/explore self-heals navigation in the same commit. Consequence: note-writers (the `vault-explorer` agent) just drop a correctly-tagged file with a lead paragraph and never touch the MOC. MOC/index titles carry a `MOC:` prefix (template + seed).
- **Stowed Claude customization** (`claude/` package, work envs only): `claude/.claude/agents/vault-explorer.md` (explores a subtopic in its own context, writes full findings to the vault, returns only a short summary + path â€” the actual fix for context-waste, since slash commands run *in* main context and can't evict) and `claude/.claude/commands/{explore,act,improve}.md`. When adding more Claude commands/agents, put them under `claude/.claude/{commands,agents}/` so they version-control and stow.

## Files that aren't stow-ed

`.stow-local-ignore` (top-level and per-package) excludes things like `install.sh`, `.git`, `.zsh_secrets`, and `.gitconfig.local` from symlinking. Adding a new file at a package root that *shouldn't* land in `$HOME` requires adding it to the relevant `.stow-local-ignore`.
Any file that is sourced in a stowed module via a direct path to the dotfiles folder should not be stowed.
